# Story 4.5-4: Add CI Lint for `EXPO_PUBLIC_*` Service Key Exposure

**Status:** done  
**Epic:** 4.5 — Retrospective Debt Resolution  
**Origin:** Epic 4 retrospective — Story 4.1 discovered service-role key in `EXPO_PUBLIC_*`

---

## Story

**As a** developer deploying the app,  
**I want** CI to fail if a service-role key is accidentally assigned to an `EXPO_PUBLIC_*` variable,  
**So that** backend-only credentials can never be bundled into the client-side JavaScript and exposed to all users.

---

## Background

### The Risk

Any variable prefixed with `EXPO_PUBLIC_` is bundled into the client-side JavaScript at build time and is visible to anyone who downloads the app or inspects the web bundle. During Epic 4 Story 4.1, the backend service-role key was discovered in the `.env` file under an `EXPO_PUBLIC_*` variable. The service-role key bypasses Row Level Security and has full database admin access. If a production web export had shipped with this variable, the key would have been publicly readable.

## Dev Notes

### Script Content

Create `apps/mobile/scripts/assert-no-service-key-exposure.sh`:

```bash
#!/bin/bash
# assert-no-service-key-exposure.sh
#
# Fails if any EXPO_PUBLIC_* variable has "service" or "service_role" in its name.
# EXPO_PUBLIC_* variables are bundled into client-side JS — backend-only credentials
# must never be assigned to them.
#
# Origin: Epic 4 retrospective (Story 4.1 finding)

set -euo pipefail

ENV_FILES=$(find apps/mobile -maxdepth 1 -name '.env*' -not -name '.env.example' 2>/dev/null || true)

if [ -z "$ENV_FILES" ]; then
  echo "✓ No .env files found — no service key exposure risk"
  exit 0
fi

FOUND=0
for f in $ENV_FILES; do
  if grep -iE 'EXPO_PUBLIC[_A-Z0-9]*(SERVICE|SERVICE_ROLE)[_A-Z0-9]*\s*=' "$f" 2>/dev/null; then
    echo "ERROR: Service-role key found in EXPO_PUBLIC variable in $f"
    FOUND=1
  fi
done

if [ "$FOUND" -eq 1 ]; then
  echo ""
  echo "EXPO_PUBLIC_* variables are bundled into client JS and visible to all users."
  echo "Move backend-only credentials to variables WITHOUT the EXPO_PUBLIC_ prefix."
  exit 1
fi

echo "✓ No service keys found in EXPO_PUBLIC variables"
exit 0
```

### CI Workflow Modification

Add a step to `.github/workflows/mobile-prod-harness-guard.yml` **before** the existing `Verify production bundle` step:

```yaml
      - name: Assert no service key in EXPO_PUBLIC variables
        working-directory: .
        run: bash apps/mobile/scripts/assert-no-service-key-exposure.sh
```

Note: `working-directory: .` (workspace root) because the script uses `apps/mobile` path prefix internally.

### Script Placement Note

The script runs from the **workspace root**, not from `apps/mobile/`. The `find apps/mobile ...` path in the script already accounts for this. The CI workflow defaults to `working-directory: apps/mobile` but this step overrides it to `.` (workspace root).

### Security Note

The script checks variable **names** containing `SERVICE` or `SERVICE_ROLE` — it does NOT scan variable **values**. This avoids false positives (e.g., an `EXPO_PUBLIC_` variable whose value happens to contain the word "service" in a sentence) while still catching the actual risk class: accidentally naming a credential variable with the public prefix.

### What a `.env.example` Should Contain

After this story, ensure `apps/mobile/.env.example` documents the correct separation:
```
# Public (safe to expose — bundled into client JS)
EXPO_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
EXPO_PUBLIC_SUPABASE_ANON_KEY=your-anon-key

# Private (backend-only — MUST NOT have EXPO_PUBLIC_ prefix)
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
BACKEND_API_URL=https://your-backend.com
```
