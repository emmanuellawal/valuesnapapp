## Story

**As a** developer modifying the AI valuation pipeline,  
**I want** a contract test that exercises `transformValuationResponse` against the real backend response shape,  
**So that** any renaming of keys between `backend/main.py` and `apps/mobile/types/transformers.ts` fails immediately instead of silently producing `undefined` in production.

## Background

### Root Cause

The transformer was originally written with wrong key names (`item_identity`, `market_data`). The bug was invisible throughout all of Epic 2 because mock mode was used exclusively. It only surfaced in Epic 3 Story 3.2 when real API calls were made. Both the Epic 2 and Epic 3 retrospectives called for a contract test. No test was ever written.

## Acceptance Criteria

### AC1: Full Response Transforms Correctly

**Given** a fixture exactly matching the backend `/api/appraise` response shape  
**When** `transformValuationResponse(fixture)` is called  
**Then** the output contains:
- `itemDetails.itemType` === `"Vintage Camera"` (from `identity.item_type`)
- `itemDetails.brand` === `"Canon"` (from `identity.brand`)
- `itemDetails.identifiers.upc` contains the UPC value (lowercased key: `upc`, mapped from `UPC`)
- `marketData.fairMarketValue` === the `valuation.fair_market_value` value
- `marketData.priceRange` is set from `valuation.price_range`
- `confidence.marketConfidence` === `"HIGH"` (from `confidence.market_confidence`)
- `valuationId` === the `valuation_id` UUID string

### AC2: Null `valuation_id` Handled

**Given** a fixture with `valuation_id: null`  
**When** `transformValuationResponse(fixture)` is called  
**Then** `output.valuationId === null`

### AC3: Null `price_range` Handled

**Given** a fixture with `valuation.price_range: null`  
**When** `transformValuationResponse(fixture)` is called  
**Then** `output.marketData.priceRange === undefined`

### AC4: Key Mismatch Causes Test Failure

**Given** the fixture uses `identity` as the primary identity key  
**When** the test runs against the real transformer code  
**Then** accessing `raw.identity` inside the transformer produces correct data  
**And** if the transformer were changed to use `raw.item_identity` instead, the test output would be `undefined` / fail

_(This AC documents the self-verifying nature of the test — it proves the test catches the Epic 3 bug class.)_

### AC5: `parseVisualCondition` Fuzzy Matching Tested

**Given** the following input values: `"LIKE_NEW"`, `"Excellent Condition"`, `"acceptable"`, `"broken"`, `"used_good"`  
**When** `transformItemDetails` is called with `visual_condition` set to each value  
**Then** the outputs are: `"used_excellent"`, `"used_excellent"`, `"used_fair"`, `"used_good"` (fallback), `"used_good"` respectively

### AC6: All Sub-Transformers Individually Tested

**Given** isolated fixture inputs for each sub-transformer  
**When** `transformIdentifiers`, `transformItemDetails`, `transformMarketData`, `transformConfidenceData` are called  
**Then** each produces correctly shaped output with camelCase keys

## Dev Notes

### Critical Implementation Notes

1. **Import path**: `import { transformValuationResponse, transformItemDetails, transformMarketData, transformConfidenceData, transformIdentifiers } from '@/types/transformers';` — `parseVisualCondition` is private (not exported); test it indirectly via `transformItemDetails`.

2. **Do NOT mock `@/types/transformers`** — other test files mock this module (e.g., `camera-guest-mode.test.tsx` line 29). This file must import and call the real functions.

3. **No React, no native modules** — this is a pure TypeScript function test. No need for `@testing-library/react-native` or `jest.setup.js` RN mocks.

4. **Key validation approach for AC4**: Include a comment in the test like:
   ```ts
   // This fixture uses the REAL backend key 'identity' (not 'item_identity').
   // If transformValuationResponse ever reverts to raw.item_identity, this test fails.
   ```

5. **UPC casing**: Backend uses `UPC` (uppercase). Transformer maps it to `upc` (lowercase). Verify this in the `transformIdentifiers` test.

6. **`no_data` status path**: Test `transformMarketData` with `status: "no_data"`, no `price_range`, no `fair_market_value` to verify the optional fields produce `undefined` (not `null`).
