# Story 5.5-7: Network Polish — Remove Tunnel Dead Weight + Frontend Retry/Backoff

Status: done

## Story

As a developer on the ValuaSnap team,
I want the mobile app to absorb transient network failures gracefully and to stop carrying dead weight from the abandoned ngrok tunnel workflow,
so that device testing is reliable under real-world Wi-Fi flakiness and the repo no longer maintains code it does not use.

## Background

Two follow-ups surfaced during the 5.5-2 party-mode review:

1. **Tunnel dead weight** — With the backend on Render Starter (no cold starts) and WSL device-testing sorted via `start:wsl` and mirrored networking (Story 5.5-2 follow-on), `expo start --tunnel` is no longer used. The `@expo/ngrok` dep, the `patch-expo-ngrok.cjs` postinstall, and the custom `expo-ngrok-v3.cjs` shim exist solely to prop up a workflow we have abandoned. Carrying them is drift liability: the next @expo/ngrok minor bump silently re-introduces a path that we know is broken on iOS (ngrok-free.dev interstitial).

2. **No retry/backoff on the appraise call** — `apps/mobile/lib/api.ts::appraise()` uses raw `fetch()` with no timeout, no retry, no backoff. Any transient network blip or upstream 502/503/504 surfaces immediately as `AppraiseError('NETWORK_ERROR')`. With Starter tier the backend itself is stable, but devices on real Wi-Fi hit transient issues regularly (captive portal handshakes, flaky carrier handoff on cellular, Cloudflare edge retries).

## Acceptance Criteria

1. `@expo/ngrok`, `apps/mobile/scripts/patch-expo-ngrok.cjs`, and `apps/mobile/scripts/expo-ngrok-v3.cjs` are deleted from the repo
2. `apps/mobile/package.json` has no `postinstall`, `patch:expo-ngrok`, `start:tunnel`, or `ios:tunnel` scripts
3. `npm install` in `apps/mobile/` completes cleanly with no reference to `@expo/ngrok`, and the resulting `package-lock.json` has no `@expo/ngrok` entry
4. `apps/mobile/lib/api.ts::appraise()` retries on network errors and on HTTP 502/503/504, with exponential backoff, capped at 2 retries (3 attempts total), and each attempt has a per-request timeout (30 s) enforced via `AbortController`
5. 4xx responses (including 429) and 2xx/3xx responses are NOT retried — retrying a 4xx is wasted work and retrying a 429 makes rate-limiting worse
6. `npm run lint` passes with zero warnings (same gate as 5.5-1)
7. `npm run test` passes; new unit tests cover: (a) retry on network error, (b) retry on 503, (c) no retry on 400/429, (d) eventual throw after max retries

## Dev Notes

### Retry/backoff design decisions

**Why not retry 4xx?** A 400 means the payload is bad. A 401 means the session is expired. A 404 means the route is gone. Retrying any of these is wasted work and latency. Special-case 429 is non-negotiable — retrying a rate-limit immediately makes the rate-limit worse.

**Why not retry 5xx broadly?** 500 means the server hit an unhandled exception. 501/505 are configuration errors. 507/508 are resource errors. Retrying these blindly risks compounding a backend problem. The narrow 502/503/504 window catches specifically: gateway-level transients that are actually recoverable (Cloudflare retrying upstream, Render service restart during auto-deploy, etc.).

**Why `AbortController` for timeout instead of `Promise.race`?** Promise-race leaves the fetch dangling (still consuming memory, still eventually resolving into nothing). `AbortController` tears down the socket. On mobile, sockets are expensive.

**Why 30 s per attempt?** Warm backend is 10.9 s for a full appraise (measured). 30 s gives a 3x safety margin. Too short = false aborts on legitimately-slow appraisals. Too long = user sees a dead app for too long before any feedback.

**Why 2 retries, not 3 or 5?** Worst-case latency with 2 retries is 30 + 1 + 30 + 3 + 30 = 94 s. Adding a third retry pushes it past 2 min, at which point the user has given up. Two retries is the sweet spot for "absorbed 99% of transients" without "torture the user".

**Idempotency caveat:** `/api/appraise` writes a valuations row on success. A retry after a successful write that dropped its response mid-flight would create a duplicate valuation. Out of scope for this story (would need a server-side idempotency-key mechanism). Documented here for the next pass.

### Why remove tunnel entirely instead of keeping it as a fallback?

Three reasons, ranked by weight:

1. **Known broken path.** ngrok-free.dev serves an HTML interstitial that Expo Go's WSS upgrade rejects as a TLS error (reproduced in Story 5.5-2). Keeping `start:tunnel` means some future dev will try it, fail, and re-open the same investigation.
2. **Dead code decays.** The `patch-expo-ngrok.cjs` postinstall exists because `@expo/ngrok` drifted against the locally-installed `ngrok` binary. That drift will recur — every minor bump of @expo/ngrok needs the patch re-verified. Deleting the whole chain deletes that maintenance tax.
3. **Better alternatives exist.** For same-Wi-Fi device testing: `npm run start:lan` (macOS/Linux) or `npm run start:wsl` (Windows/WSL). For off-network testing: build an EAS update channel and install on the device — that's the production path Epic 6+ will use anyway.

### Testing the retry logic

Jest fake timers are required because backoff uses `setTimeout`. Pattern:

```typescript
jest.useFakeTimers();
// mock fetch to reject twice, resolve third time
const p = appraise('image-data', 'session-id');
await jest.advanceTimersByTimeAsync(1000); // drain first backoff
await jest.advanceTimersByTimeAsync(3000); // drain second backoff
await expect(p).resolves.toBeDefined();
```

### Out of scope (logged for future)

- **CORS tightening in `backend/main.py`** — drop the `["*"]` fallback and `allow_credentials=True`. Hygiene, not urgent; deferred to `epic-7-12-configure-production-monitoring` or a dedicated hardening story.
- **Server-side idempotency keys** for `/api/appraise` — needed before this retry logic is safe at scale. Logged for Epic 6 or later.
- **Keep-warm ping** — moot on Starter tier (no sleep). If we ever downgrade to free tier, revisit.

### Completion Notes List

- **Ngrok purge — scope wider than expected.** `npm install` removed 30 transitive packages when `@expo/ngrok` came out, which is a solid measure of how much dead weight the tunnel workflow was carrying.
- **`fetchWithRetry` exported, not private.** Exporting the helper let the unit tests hit retry paths directly with a tiny `backoffBaseMs: 1` override, avoiding the 4-plus-second real-timer wait that the `appraise()` path needs. Net: the retry matrix runs in under 50 ms.
- **Existing `throws NETWORK_ERROR` test had to change.** The old test called `appraise()` twice back-to-back; with retries now in the loop that would be 6 fetch attempts with 8+ s of real backoff, exceeding Jest's default 5 s timeout. Rewrote to use fake timers and verify call count == 3 (1 initial + 2 retries).
- **Deliberate omissions (logged in Dev Notes):** no CORS tightening in backend (hygiene work, deferred), no idempotency keys yet (would need server changes; documented as a caveat).
- **Starter-tier Render means we're not seeing 502/503/504 in practice right now.** The retry logic is insurance for device-side flakiness (captive portals, cellular handoff, Cloudflare edge), not a backend workaround.
