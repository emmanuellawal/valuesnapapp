## Dev Notes

### Critical: Module Path — Why Root Directory Must Be `.` Not `backend/`

`backend/main.py` uses absolute package imports:
```python
from backend.cache import get_cache_stats, get_supabase, get_user_supabase
from backend.config import allowed_origins, settings
```

This means Python must be able to find `backend` as a package, which requires the working directory to be the **repo root** (`/`), not inside `backend/`. If Render's root directory is set to `backend/`, these imports fail with `ModuleNotFoundError: No module named 'backend'` and the service crashes at startup.

**Correct Render settings:**
```
Root Directory:   .   (repo root, or leave blank)
Build Command:    pip install -r backend/requirements.txt
Start Command:    uvicorn backend.main:app --host 0.0.0.0 --port $PORT
```

### Environment Variables — Exact Names from `backend/config.py`

`pydantic-settings` maps field names directly to env var names (uppercase). The epic plan's variable names (`EBAY_APP_ID`, `EBAY_CERT_ID`) are **wrong** — they match nothing in `config.py`. Use these exact names:

| Env Var (Render) | `config.py` field | Value source |
|---|---|---|
| `OPENAI_API_KEY` | `openai_api_key` | `backend/.env` |
| `EBAY_PROD_APP_ID` | `ebay_prod_app_id` | `backend/.env` |
| `EBAY_PROD_CERT_ID` | `ebay_prod_cert_id` | `backend/.env` |
| `EBAY_USE_SANDBOX` | `ebay_use_sandbox` | `false` |
| `USE_MOCK` | `use_mock` | `false` |
| `SUPABASE_URL` | `supabase_url` | `backend/.env` |
| `SUPABASE_SERVICE_KEY` | `supabase_service_key` | `backend/.env` |
| `CORS_ORIGINS` | `cors_origins` | See CORS section below |

**Do NOT set `SUPABASE_ANON_KEY`** in the backend — that key belongs to the frontend only (`EXPO_PUBLIC_SUPABASE_ANON_KEY` in `apps/mobile/.env`). The backend uses only the service key.

**Do NOT use `EBAY_APP_ID` or `EBAY_CERT_ID`** — those field names do not exist in `config.py` and will be silently ignored, leaving eBay calls broken.

### CORS Configuration

CORS origins are read from `CORS_ORIGINS` as a comma-separated string and parsed by `parse_origins()` in `config.py`. When empty, `main.py` falls back to `allow_origins=["*"]`.

**Critical:** Starlette's `CORSMiddleware` performs **exact string match** on origins — it does NOT support wildcard subdomains like `https://*.expo.dev`. Any wildcard entry is silently ignored and never matches an incoming `Origin` header. Wildcard subdomain matching requires `allow_origin_regex`, which the current backend does NOT wire up.

**Recommended for this story:** leave `CORS_ORIGINS` UNSET on Render. The fallback `["*"]` allows any origin — acceptable for a pre-production deployment used for device and browser smoke tests. Tightening CORS to exact origins (or switching to `allow_origin_regex`) is a separate hardening story before production.

**If you insist on setting `CORS_ORIGINS`:** list only exact origins — no wildcards. Example values that actually work:
```
http://localhost:8083,http://localhost:19006,https://valuesnap.app
```

**CORS vs device testing:** CORS only applies to browser-issued requests. Native clients (Expo Go, iOS/Android apps) do not send an `Origin` header and bypass CORS entirely. AC 2 (device test) therefore does not validate CORS — only AC 6 does.

### Free-Tier Sleep Warning

Render free-tier services spin down after 15 minutes of inactivity. The first request after a cold start takes 30–60 seconds. This is acceptable for this story's scope (real-device testing, not production SLA). If sleep causes timeout failures in the end-to-end device test, upgrade to the Starter tier ($7/month) and note it in the completion notes.

### Auto-Deploy Behavior

Render auto-deploys on every push to the branch configured in the service (default: `main`). This means:
- Any future commit to `main` redeploys the backend automatically
- A broken commit takes the service down until the next successful deploy
- The CI pipeline from Story 5.5-6 should catch lint/test failures before merge — but it does NOT gate Render deploys. A PR that is merged with CI red (hotfix bypass, for example) will still trigger a deploy.

Document the Render URL and this auto-deploy behavior in `docs/deployment/README.md` so the team understands the coupling.

### Python version pinning — what actually works on Render

`runtime.txt` is a **Heroku** convention. Render does NOT read it. Both the first and second failed deploys installed Python 3.14 (visible as `cp314` wheels) because `runtime.txt` was silently ignored in both locations.

Render respects **two** mechanisms:
1. `.python-version` file at the **repo root** (format: `3.11.9`)
2. `PYTHON_VERSION` environment variable (set in dashboard or `render.yaml`)

This story uses both: `.python-version` at repo root + `PYTHON_VERSION` in `render.yaml`.

### `backend/config.py` — pydantic-settings v2 modern syntax

The original `Settings` class used a deprecated pydantic v1-style inner `class Config` with `import os` inside the class body. On April 20, 2026 (the day before the second failed deploy), `pydantic-settings 2.14.0` was released. The interaction between the deprecated `class Config` pattern, Python 3.14, and the new pydantic-settings release is the most likely cause of the startup crash. Migrated to `model_config = SettingsConfigDict(...)` — fully supported in pydantic-settings 2.x and eliminates the deprecated pattern entirely.

### Dashboard Root Directory check — critical

If the Render service was created with Root Directory = `backend/`, the start command `uvicorn backend.main:app` would fail with `ModuleNotFoundError` because Python would look for `backend/backend/main.py`. Verify Root Directory = `.` in the Render dashboard before the next deploy.
