# Story 5.5-6: Add CI Lint Pipeline

Status: ready-for-dev

## Story

As a developer on the ValuaSnap team,
I want a GitHub Actions workflow that runs `npm run lint` and `npm test` on every pull request to `main`,
so that no commit can be merged without passing the debt gates that Story 5.5-1 installed ‚Äî closing the gap between a local lint script and an enforced process gate.

## Background

Story 5.5-1 installed `npm run lint` (`expo lint --max-warnings 0`) and extended the frontend review checklist. During the code review for 5.5-1, finding M4 was logged:

> The script now exists locally; the actual enforcement on merge does not. Any dev can merge a PR to `main` without running lint. The gate is local-only.

`docs/architecture.md` L524‚Äì571 already specifies the exact GitHub Actions YAML for CI/CD (`test-backend` + `test-frontend` jobs, triggered on PRs to `main`). This story implements the `test-frontend` job from that spec.

**Scope:** `test-frontend` job only. `test-backend`, `deploy-staging`, and `deploy-production` jobs are out of scope ‚Äî they depend on Render (Story 5.5-2) and are deferred until 5.5-2 is done.

## Acceptance Criteria

1. `.github/workflows/ci.yml` exists in the repository and is committed to git
2. On every pull request targeting `main`, the `test-frontend` job runs automatically in GitHub Actions
3. The `test-frontend` job runs `npm ci`, `npm run lint`, and `npm test` from `apps/mobile/` ‚Äî the exact commands that pass locally after 5.5-1
4. If `npm run lint` exits non-zero (any warning or error), the CI job fails and the PR is blocked
5. If `npm test` exits non-zero, the CI job fails and the PR is blocked
6. The workflow file uses Node.js 20 and `actions/checkout@v4` / `actions/setup-node@v4` (matches architecture spec)
7. The CI job completes successfully on the current `main` HEAD (verified via a test run or dry-run)

## Dev Notes

### Task A ‚Äî Workflow file

**Reference architecture:** `docs/architecture.md` L524‚Äì571 contains the authoritative YAML. The `test-frontend` job to implement is:

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test-frontend:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: apps/mobile
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run lint
      - run: npm run test
```

**Key constraint:** `npm run lint` is `expo lint --max-warnings 0` (set by 5.5-1). Any warning in the codebase will fail CI. This is intentional ‚Äî it is the debt gate.

**Why `npm ci` instead of `npm install`:** `npm ci` respects the lockfile exactly and fails if the lockfile is out of sync with `package.json`. It is the correct command for CI.

**Working directory:** Use the `defaults.run.working-directory` block at the job level rather than repeating `working-directory` on each `run` step ‚Äî cleaner, DRY, and matches the convention for monorepos.

**Name the workflow `CI` not `CI/CD`:** The architecture spec calls it `CI/CD` but this file only implements the test jobs. Rename to `CI` to accurately reflect scope; `CD` will be wired in a future story once Render (5.5-2) is live and the deploy steps have a real URL.

**Out-of-scope jobs** (do not add to this file):
- `test-backend` ‚Äî requires `backend/` Python setup; deferred
- `deploy-staging` / `deploy-production` ‚Äî require Render URL from 5.5-2; deferred

These can be added to `ci.yml` in future stories without conflict ‚Äî GitHub Actions jobs are additive.

### Task B ‚Äî Smoke test

No new unit tests are required for this story ‚Äî the workflow is YAML, not application code. The validation is:
1. Local command passes ‚Üí CI will pass on same code
2. YAML parses without error ‚Üí workflow will not fail with a syntax error on push

**`act` note:** `act` (local GitHub Actions runner) is not required. If it's installed, `act pull_request` is a useful dry-run. If not, skip ‚Äî the local command smoke test is sufficient.

### Project Structure Notes

- `.github/workflows/ci.yml` ‚Äî new file; must be committed (it only takes effect once pushed to GitHub)
- No changes to `apps/mobile/` code ‚Äî this story is purely infrastructure config
- The workflow runs from the repo root; `working-directory: apps/mobile` scopes the npm commands to the correct workspace
