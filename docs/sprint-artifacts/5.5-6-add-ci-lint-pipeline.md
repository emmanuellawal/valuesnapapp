# Story 5.5-6: Add CI Lint Pipeline

Status: done

## Story

As a developer on the ValuaSnap team,
I want a GitHub Actions workflow that runs `npm run lint` and `npm test` on every pull request to `main`,
so that no commit can be merged without passing the debt gates that Story 5.5-1 installed — closing the gap between a local lint script and an enforced process gate.

## Background

Story 5.5-1 installed `npm run lint` (`expo lint --max-warnings 0`) and extended the frontend review checklist. During the code review for 5.5-1, finding M4 was logged:

> The script now exists locally; the actual enforcement on merge does not. Any dev can merge a PR to `main` without running lint. The gate is local-only.

`docs/architecture.md` L524–571 already specifies the exact GitHub Actions YAML for CI/CD (`test-backend` + `test-frontend` jobs, triggered on PRs to `main`). This story implements the `test-frontend` job from that spec.

**Scope:** `test-frontend` job only. `test-backend`, `deploy-staging`, and `deploy-production` jobs are out of scope — they depend on Render (Story 5.5-2) and are deferred until 5.5-2 is done.

## Acceptance Criteria

1. `.github/workflows/ci.yml` exists in the repository and is committed to git
2. On every pull request targeting `main`, the `test-frontend` job runs automatically in GitHub Actions
3. The `test-frontend` job runs `npm ci`, `npm run lint`, and `npm run test:ci` from `apps/mobile/` — the commands that gate every merge
4. If `npm run lint` exits non-zero, the CI job fails AND `main` branch protection blocks merge (a red check alone is not enough — see Task C)
5. If `npm run test:ci` exits non-zero, the CI job fails AND branch protection blocks merge
6. The workflow file uses Node.js 20, `actions/checkout@v4`, and `actions/setup-node@v4` with `cache: 'npm'` (matches architecture spec, with caching added for speed)
7. Local smoke verification passes on the current repo HEAD (`npm ci && npm run lint && npm run test:ci` from `apps/mobile/` all exit 0). Real CI run on GitHub is first observed on the PR that introduces this workflow and is tracked under Task C verification, not AC7 — AC7 is intentionally scoped to what is verifiable before push.
8. Branch protection rule on `main` requires the `test-frontend` check to pass before merge (configured via `gh api` or GitHub UI; verified in Task C)

## Dev Notes

### Task A — Authoritative workflow YAML

This is the single source of truth. Task A implements exactly this — no additions, no omissions.

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

permissions:
  contents: read

jobs:
  test-frontend:
    runs-on: ubuntu-latest
    timeout-minutes: 15
    defaults:
      run:
        working-directory: apps/mobile
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version-file: apps/mobile/.nvmrc
          cache: 'npm'
          cache-dependency-path: apps/mobile/package-lock.json
      - run: npm ci
      - run: npm run lint
      - run: npm run test:ci
```

**Deltas from `docs/architecture.md` L524–571 (explained):**

1. **Scope reduced to `test-frontend` only.** `test-backend`, `deploy-staging`, `deploy-production` are deferred — they require Render (5.5-2) to be live. GitHub Actions jobs are additive, so future stories append without conflict.
2. **Workflow name `CI` not `CI/CD`.** Spec name overstates current scope; CD will land in a follow-up story.
3. **`push` trigger narrowed to `[main]` only.** Spec includes `develop`, but this repo has no `develop` branch (verified via `git branch -a`). Adding a dead trigger is clutter.
4. **`defaults.run.working-directory: apps/mobile` at the job level.** Spec omits this; without it, every `run` step would execute at repo root where there is no `package.json`. DRY and idiomatic for monorepos.
5. **`cache: 'npm'` + `cache-dependency-path` added.** Spec omits caching. Without it, every CI run pays ~2–3 min cold-install cost AND fails on transient npm registry outages. `setup-node@v4` caches keyed on `package-lock.json` SHA; cold runs stay correct, warm runs finish in ~20–30 s.
6. **`concurrency` block added.** Cancels superseded runs when multiple commits land in rapid succession — saves CI minutes and surfaces the latest-commit result faster.
7. **`npm run test:ci` instead of `npm run test`.** See Task B — `--ci` flag disables interactive snapshot writes and other CI-hostile defaults.
8. **`permissions: contents: read` pinned.** GitHub best practice since 2023. Without an explicit block the workflow inherits repo default token scope; for a read-only lint/test job we don't want write permissions on the `GITHUB_TOKEN`.
9. **`timeout-minutes: 15` on the job.** Default job timeout is 360 minutes. A flaky jest hang or wedged `npm ci` would silently eat Actions minutes until then. Local run is ~12s, so 15 min is ~70x headroom — tight enough to fail fast, loose enough to never false-kill.
10. **`node-version-file: apps/mobile/.nvmrc` instead of `node-version: '20'`.** Pinning the major version (`'20'`) floats to whatever patch `setup-node@v4` resolves that day. Expo packages currently require `>=20.19.4`; resolving to an older patch would hit `EBADENGINE` failures. `.nvmrc` (committed in `apps/mobile/`) keeps CI and local dev on the same exact patch version deterministically.

**Key constraints the dev must understand:**

- `npm run lint` is `expo lint --max-warnings 0` (set by 5.5-1). Any warning anywhere in `apps/mobile/` fails CI — even in a file your PR didn't touch. This is intentional. This is the debt gate. Do not relax it.
- `npm ci` is correct for CI (not `npm install`): it respects the lockfile exactly and fails on drift.
- The workflow runs from repo root; `working-directory: apps/mobile` scopes all `run` steps. Do not use paths-based filters on the trigger (e.g., `paths: apps/mobile/**`) in this story — that's an optimization for later, after branch protection is live, to avoid CI skipping the required check entirely on unrelated changes.

### Task B — Why `test:ci`

`jest` with no flags is optimized for a developer's interactive terminal: it writes missing snapshots, prints a progress UI, and can silently pick up watch-mode defaults. In CI:

- **`--ci`** disables watch mode, treats missing snapshots as failures (instead of writing them and passing), and uses non-interactive output.
- **`--runInBand`** forces serial execution. The test suite has shown flakiness under default parallelism (see 5.5-1 retrospective). Serial execution is slower but deterministic — worth it for gate reliability.

Add `"test:ci": "jest --ci --runInBand"` to `apps/mobile/package.json#scripts`. Leave `"test": "jest"` alone — devs still want fast interactive runs locally.

### Task C — Branch protection (the gate that actually blocks merges)

**Why this task exists:** A red X on a PR does not block merge by itself. GitHub only blocks merges when branch protection requires a named status check. Shipping `ci.yml` without this step ships a visible-but-unenforced gate — the exact failure mode 5.5-1 was created to prevent. This is the disaster this story is meant to close.

**Order of operations matters:**
1. Merge `ci.yml` to `main` first.
2. Wait for at least one `test-frontend` run to complete so GitHub registers the check name in the repo's check history.
3. Then configure branch protection — GitHub's UI/API will not let you require a check name it has never seen.

**Preferred: `gh` CLI.** Requires `gh auth status` to show admin on the repo:

```bash
gh api -X PUT "repos/{owner}/{repo}/branches/main/protection" \
  --input - <<'EOF'
{
  "required_status_checks": { "strict": true, "contexts": ["test-frontend"] },
  "enforce_admins": false,
  "required_pull_request_reviews": null,
  "restrictions": null
}
EOF
```

Adjust `enforce_admins` / `required_pull_request_reviews` to match repo policy — the critical field is `required_status_checks.contexts`.

**Fallback: GitHub UI.** Settings → Branches → Add branch protection rule → Branch name pattern: `main` → Require status checks to pass before merging → Require branches to be up to date before merging → Search and select `test-frontend` → Save.

**Verification:** Open a throwaway branch, introduce a deliberate lint failure (e.g., an unused import), push, open a PR. The merge button must be disabled with the message "Required statuses must pass before merging." Revert the bad commit; the merge button re-enables once CI turns green.
