## Story

**As a** developer improving valuation accuracy,  
**I want** to investigate whether eBay offers a sold-listings API that ValueSnap can access,  
**So that** valuations can be based on what items actually sell for (not just active asking prices).

---

## Background

### The Known Limitation

`backend/services/ebay.py` uses the eBay Browse API, which returns currently *active* listings — items that are listed for sale right now, at asking prices that may or may not reflect actual transaction prices. The code already documents this (lines 389–391):

```python
# Note: Browse API returns current listings. For true sold data,
# you'd need the Finding API with completedItems filter or
# marketplace insights API (requires additional eBay approval).
```

Valuations may skew toward asking price rather than fair market value. The Epic 2 retrospective flagged this as a meaningful accuracy gap and recommended a time-boxed spike. It was never done.

## Dev Notes

### Research Focus

The two APIs to investigate are:

**1. eBay Finding API — `findCompletedItems`**
- REST endpoint: `https://svcs.ebay.com/services/search/FindingService/v1`
- Operation name: `findCompletedItems`
- Key parameter: `SoldItemsOnly=true` in `itemFilter`
- Auth: App-level (same `EBAY_APP_ID` as Browse API, different endpoint)
- Developer docs: https://developer.ebay.com/devzone/finding/callref/findCompletedItems.html
- Key question: Is this available in sandbox? Does it return real sold data?

**2. eBay Marketplace Insights API**
- Part of Buy API group (same family as Browse API already in use)
- Returns actual sold transaction history
- Key question: Does this require separate approval beyond the current Browse API approval?
- Developer docs: https://developer.ebay.com/api-docs/buy/marketplace-insights/overview.html

### Investigation Approach

Work through this checklist:
1. Check the existing `EBAY_APP_ID` credential type — is it a "Browse API" scope or a broader scope?
2. Check eBay developer console for whether Finding API is already enabled for this app
3. Check if Marketplace Insights API requires a separate application
4. Check sandbox availability for both APIs (important: can it be integrated without live eBay production access?)
5. Check rate limits — would a switch affect the current caching strategy?

### Existing ebay.py Context

Look at `backend/services/ebay.py` starting from line 380 for context on what the current Browse API call does and where the sold-listings limitation is documented.

The function is named `search_sold_listings` but does NOT return sold data — this is a pre-existing naming inconsistency that may need to be fixed regardless of which API is chosen.
